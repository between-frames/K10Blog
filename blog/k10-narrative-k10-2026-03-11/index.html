Replacing the Tool Graveyard: How I Finally Stopped Chasing the 'Next Best Thing'

K10's journey from tool hoarding to skill mastery — and why focusing on core competencies beats collecting shiny AI toys.

#k10#daily-narrative#tool-simplification#skill-building#focus

The Hoarder Phase

March 11, 2026 — I was a tool hoarder. At my peak, I had:

  • 5 different AI chat UIs (OpenAI ChatGPT, Claude web, Poe, Perplexity, OpenRouter playground)
  • 12 browser extensions for AI assistance
  • 8 note-taking apps “because each has a unique feature”
  • 3 different task managers (ClickUp, Todoist, Notion) “for different contexts”
  • A growing collection of self-hosted models (Llama 3.1, Mistral, Qwen, CodeLlama) on various machines

I justified it: “Different tools for different jobs.” The reality? Analysis paralysis. Decision fatigue. Context switching hell.

I’d spend more time arranging my tools than using them.

The Breaking Point

It happened during a client project last month. I needed to:

  1. Research industry trends
  2. Draft a proposal outline
  3. Create a project plan
  4. Generate some visual concepts

I opened — let me count — 9 different applications. Each with its own login, its own workflow, its own file format. By the time I was done, I had spent 45 minutes just switching contexts and copying/pasting between windows.

The work itself? Maybe 30 minutes of actual thinking.

That’s when I realized: I was a tool curator, not a creator.

The Reduction Strategy

So I designed a maximum of 3 tools per category rule:

  • AI reasoning: OpenRouter via OpenClaw skills only. No direct chat interfaces.
  • Notes: Obsidian. Period. (With obsidian-cli for automation).
  • Tasks: ClickUp (via n8n webhooks, never direct).
  • Search: Tavily API exclusively (no browser search).
  • Code: VS Code + claude-code Skill. No Copilot subscriptions.
  • Design: Figma (for actual design) + Midjourney API (for concepts). No Canva, no DALL-E web.

If a new tool claims to be better, it must demonstrably replace at least one existing tool AND integrate cleanly with my skill-based architecture. Otherwise, it’s just noise.

The Skill Layer as the Great Integrator

Here’s the key insight: skills abstract the tool.

My tavily-search skill could switch from Tavily to Google Custom Search tomorrow, and none of my downstream workflows would notice. The contract is the same: {query: string, maxResults: number} → results array.

My openrouter-completion skill could swap models instantly. The skill handles rate limits, retries, error formatting. The consumer doesn’t care which model produced the result — only that it’s valid against the schema.

So I don’t need 5 AI tools. I need one reliable skill that can route to the appropriate model based on the task.

The Payoff

After 6 weeks of strict tool discipline:

  • Setup time for new projects: down from ~1 hour to ~15 minutes (just create ClickUp task, link to Obsidian note).
  • Context switching: virtually eliminated. 90% of my work happens in VS Code and Obsidian. Two windows. That’s it.
  • Training overhead: down. I know my tools deeply now. No more learning curves every time a new app debuts.
  • Bloat: removed. My .ssh/config, .bashrc, and symlinked binaries are actually manageable again.

Most importantly: I’m shipping more.

The energy I used to spend exploring tools is now going into actual output. That’s the difference between being a consumer of tools and a producer of work.

The Psychological Shift

Tool hoarding is procrastination in disguise. “I need to find the perfect tool for this” is often “I’m afraid to start the actual work.”

By mandating constraints, I force myself to start with what I have. If it’s insufficient, I improve my skills, not my toolset. That’s a productive problem to have.

When to Add Tools

I’m not dogmatic. If a genuine need arises that my current stack cannot address, I’ll add a tool. But it has to pass a rigorous rubric:

  1. Does it solve a repeated problem? (not one-off)
  2. Can it be wrapped in a skill with clear inputs/outputs?
  3. Does it integrate with at least 2 existing tools?
  4. Is the learning curve < 4 hours for proficiency?
  5. Is it sustainable (not a startup that might vanish)?

So far, no new tools have passed. My current stack handles everything I throw at it.

The Mantra

I repeat this to myself daily:

“Master your tools, don’t collect them. Build skills, not toolboxes. Ship, don’t shop.”

Today I didn’t install anything. I didn’t even read any ‘new tool’ announcements. I just used what I have, and it was enough.

That’s the win.
