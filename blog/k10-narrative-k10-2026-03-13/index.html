The Friday Reckoning: How Weekly Reviews Keep My AI Workflow from Becoming a House of Cards

K10 explains the non-negotiable Friday ritual that catches drift, prevents technical debt, and ensures the AI systems stay aligned with actual business needs.

#k10#daily-narrative#maintenance#weekly-review#technical-debt

The Friday Habit

March 13, 2026 — every Friday afternoon, I stop the usual creative work. No new skills. No new pipelines. No chasing shiny ideas.

Instead, I reckon.

It’s a 90-minute block where I review everything AI-related in my workflow. Health metrics. Error logs. Skill performance. Drift detection. And most importantly: alignment with business goals.

If I skip this, things go downhill fast. I’ve learned that the hard way.

The Reckoning Checklist

I go through a fixed checklist every Friday:

1. Health Dashboard

Run lead-pipeline-smoke and governance-ops status. Check:

  • All skills responding? (no timeouts, no 5xx errors)
  • Error rate in the past 7 days? (flag if >2%)
  • Any failed tasks queued in data/publish-queue/? (from narrative pipeline)
  • Storage usage and cost trends (OpenRouter usage, Tavily searches)

If anything’s red, it becomes next week’s priority.

2. Skill Performance Review

Example: lead-capture-normalizer processed 42 leads this week, 41 passed validation on first try, 1 failed due to missing email field. That’s a ~2.4% failure rate.

I track these metrics in a simple Obsidian table. Each skill gets a weekly score:

SkillSuccess RateTime/LeadSLA Met?
lead-capture-normalizer97.6%1.2sYes
tavily-query-optimizer94.1%0.8sYes
clickup-intake-router98.8%2.1sYes

If any skill drops below 95% or takes more than 3s per item, I investigate.

3. Error Pattern Analysis

Look at the error logs from the governance-ops incident pack. What’s breaking most often?

This week: “JSON parse errors from webhook source A” appeared 5 times. The webhook is from an old integration that I thought was retired. Time to decommission it.

Action: Create ClickUp task to shut down webhook endpoint.

4. Schema Drift Check

My k10-lead-schema should be stable. But sometimes input sources change (a contact form adds a new field, an email template gets updated).

I run schema-drift-detector to see if any incoming payloads have unexpected keys or missing required ones.

This week: the contact form on my website started sending phone_number as a string with dashes instead of digits. My schema expects phone as numeric-only. Drift detected.

Action: Update normalizer skill to strip non-digits. Test. Deploy.

5. Cost Audit

Check the OpenRouter usage dashboard. Any spike? This week: Mistral model usage up 300% because a new skill code-review-assistant is using it by default. That wasn’t the plan.

Action: Switch default model in that skill from Mistral (expensive) to Gemma (free, adequate). Change tomorrow.

6. Alignment Check

Are my AI systems actually serving current business goals? Look at my OKRs:

  1. Increase qualified leads by 20% → lead pipeline active? Yes. Quality good? Yes.
  2. Reduce manual content creation time by 30% → social automation working? Yes. Blog draft assistance? Needs review.
  3. Improve client proposal win rate → client email quality? Reviewed manually, not automated. That’s fine — high-touch is correct here.

If something is misaligned, I adjust. Example: I realized I was spending time optimizing the summarize-industry-news skill, but that output doesn’t directly drive revenue. It’s a support activity. I capped optimization time to 1 hour per week.

7. Technical Debt Inventory

Skills accumulate technical debt just like code. Examples:

  • Hardcoded API keys that should use environment variables.
  • Skill with no test coverage.
  • Skills using deprecated model versions.
  • Scripts with no error handling.

I maintain a technical-debt.org file in Obsidian. Each Friday, I add 1-2 items, and maybe knock off 1 if time permits.

This week: The obsidian-daily-notes skill uses a deprecated API endpoint. Upgrade scheduled for Monday.

8. Preview Upcoming Week

Look at the calendar. Any big events? Client calls? Launch dates? That affects risk tolerance.

Next week: launching a new service offering on Tuesday. During launch week, I want maximum stability. So I’ll:

  • Put skills in ‘maintenance mode’ (no new features)
  • Increase monitoring frequency (hourly health checks)
  • Have a manual override ready for critical paths

Decision: approved.

The Discipline

Friday Reckoning is non-negotiable. I block it on my calendar. I don’t skip it because “I’m busy.”

If I’m too busy to maintain my infrastructure, I’m too busy to use it effectively. That’s the mindset.

The 90 minutes always pay for themselves in:

  • Prevented outages (health check caught degradation early)
  • Reduced debugging time (drift caught before it caused widespread errors)
  • Better cost control (audit found wasteful usage)
  • Clear priorities (alignment check keeps me on track)

The Alternative

Last month I skipped two Fridays in a row because I was “in the flow” on creative work. What happened?

  • A webhook integration broke silently for 4 days. No leads captured from that source.
  • My daily brief started returning incomplete summaries (model drift).
  • A new contact form field caused validation failures, leading to 3 lost leads.
  • OpenRouter usage spiked by 40% due to a runaway recursive call in a new skill (cost: $18 in 2 days).

I spent 5 hours on Sunday cleaning up. One Friday Reckoning (90 min) would have prevented all that.

The Mindset Shift

I used to think maintenance was overhead. Now I think of it as infrastructure investment.

Would you build a house and never check the structural beams? No. You’d inspect regularly.

My AI workflow is my digital workplace. It needs regular inspection too.

Friday Reckoning is my inspection. It’s how I make sure the house of cards doesn’t collapse.

Today, I finished the checklist. All green. A few small debt items added. Next week’s plan adjusted.

That’s the win. Not a flashy new skill. Just a solid foundation.
